RAII

在讲解智能指针前，我们先了解RAII(Resource acquisition is initialization)的概念（资源构造即初始化？），简单的说就是利用类的构造函数管理资源的创建，析构函数管理资源的销毁。由于对象执行到其生存域之外时自动调用析构函数，因此我们自然联想到是否可以用一个专门的类管理指针？答案是可以的。我们先粗略实现一下这个想法：

Bingo！看来我们的想法是行得通的！指针“自动”释放了！让我们理一理思绪，上面发生了什么，首先我们动态创建了资源Resource，并将该资源的指针作为参数传给了智能指针res，作为栈对象res，当程序执行到其生存域外时，对象res销毁调用析构函数，在析构函数中释放其成员对象m_ptr，即资源Resource，这样就初步完成了我们的智能指针。等等，为什么是初步？让我们考虑下面的情况：

当使用res1初始化res2时，由于我们没有定义拷贝构造函数，C++提供了默认拷贝构造函数，通过浅拷贝，res1和res2的m_ptr指向了同一资源Resource，当程序运行结束时，res1和res2都会被销毁，当res1销毁同时的释放了Resource，这时res2的成员m_ptr成为了悬空指针，res2再销毁时程序崩溃。同样的情况还有：

res作为参数传递给函数func时，发生了浅拷贝生成临时对象，当函数func执行结束，临时对象销毁并释放指针m_ptr，外层函数main运行结束时，res销毁也会释放指针m_ptr，由于此时m_ptr已经释放，程序崩溃！

move

这里我们如何返回res？指针or引用？引用首先排除引用，func函数结束时，局部变量res销毁，返回的引用中m_ptr为悬空指针；返回Smart_ptr1的指针？这时会产生鸡生蛋，蛋生鸡的问题。

如何解决上面的问题，我们很自然地联想到自定义拷贝构造函数和赋值构造函数，但是深拷贝代价太高！提一个问题，把大象放到冰箱需要几步？三步：1.打开冰箱；2.把大象放进冰箱；3.关上冰箱。再提一个问题，如何把A冰箱的大象放到B冰箱，我们自然的会想到：打开A冰箱把大象取出来，再打开B冰箱把大象放进去，没有人会这样做：通过等离子销毁器销毁A冰箱中的大象，再通过等离子创造器创造B冰箱的中的大象。但是C++中深拷贝就是这样做的！

我们该如何解决？在上面的例子中，返回res时，我们可以确定临时对象res不再使用，因此能不能将res的资源通过某种方式转移给外部，这就是move的理念！

输出：

Resource Constructor
Copy Constructor
Resource Destructor

我们通过定义具有移动语义的拷贝构造函数和赋值构造函数，在func运行结束时，通过资源的转移解决了悬空指针的问题，同时相比于深拷贝，这样效率更高！

std::auto_ptr

C++98中引入了std::auto_ptr，它的原理类似于上面的Smart_ptr2，但是Smart_ptr2有很多缺陷。正如上面的例子中，我们非常确定作为函数func返回值的res临时对象不再被使用，资源的转移不会产生问题，但是如果Smart_ptr2对象作为参数传给函数时，其资源通过拷贝构造函数转移给了函数的临时对象，这时外部的Smart_ptr2资源为空，再次通过外部Smart_ptr2使用该资源时，程序崩溃！

因此，在C++11中不推荐使用std::auto_ptr。
